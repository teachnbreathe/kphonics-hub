# Claude Code 실행 프롬프트 — GitHub 업데이트 + 사이트 배포

> Cursor 터미널의 `claude` 명령에 그대로 붙여넣으세요. 이 프로젝트 폴더(`k-phonics-adventure`) 안에서 실행해야 합니다.

---

지금까지 업데이트된 K-Phonics OPS™ Adventure 웹앱 파일들을 GitHub에 커밋·푸시하고, GitHub Pages로 사이트에 배포해줘. 아래 순서로 진행해줘.

## 1. 현재 상태 확인

- `git status`로 변경된 파일 목록을 먼저 보여줘.
- `git remote -v`로 원격 저장소가 연결돼 있는지 확인해줘.
  - **연결돼 있으면** → 그 원격 저장소 그대로 사용.
  - **연결 안 돼 있으면** → 진행하지 말고 나한테 먼저 물어봐줘(새 GitHub 저장소를 만들지, 기존 저장소 URL을 알려줄지 확인 필요). 임의로 새 저장소를 만들지 마.
- `git log --oneline -5`로 최근 커밋 이력도 같이 보여줘 — 지금 브랜치가 뭔지, 마지막 배포가 언제였는지 감 잡게.

## 2. 변경 파일 확인 및 커밋

- 바뀐 파일들을 훑어보고(`git diff --stat`), 아래처럼 논리적 단위로 나눠서 커밋해줘. 한 덩어리로 뭉쳐서 커밋하지 말 것 — 나중에 되돌리기 쉽게.
  - 예시 단위: `js/data.js` + `js/game.js` (기능 변경), `css/style.css`(스타일), `SPEC.md` + `K-Phonics_OPS_Adventure_지도체계_마스터표.md`(문서), 기타 신규 문서 파일들.
- 커밋 메시지는 한글로, 무엇을·왜 바꿨는지 한 줄 요약 + 필요하면 본문에 bullet로 detail. 예:
  ```
  feat: 사운드박스 시각화 + 유창성 스테이지 + 사이트워드 3그룹 배치

  - 세그멘팅/블렌딩 사운드박스(onset/vowel/coda) 시각화 추가
  - 문장 피라미드 읽기(efl-fluency-1) 필수 스테이지 구현
  - Chapter1 사이트워드 Dolch 기준 3그룹 재배치(efl-2/4/6 직후)
  - old·want 사이트워드 분류 정정(파닉스 규칙목록 PDF 대조)
  ```
- 커밋 전에 `.gitignore`에 없는 스크래치/임시 파일(예: 검증용 스크립트, `.tmp` 등)이 같이 올라가지 않는지 확인해줘.

## 3. 푸시

- 원격 저장소의 기본 브랜치(main 또는 master, `git branch -a`로 확인)에 푸시해줘.
- 푸시 전에 `git pull --rebase`로 원격에 내가 모르는 변경사항이 없는지 먼저 확인하고, 충돌이 있으면 임의로 해결하지 말고 나한테 보여줘.

## 4. GitHub Pages 배포 확인

- 이 저장소가 이미 GitHub Pages로 서빙되고 있는지 확인해줘(`gh repo view --json` 또는 저장소 Settings → Pages 상태 — `gh` CLI가 설치돼 있으면 활용).
- 이미 Pages가 켜져 있다면 → 브랜치/폴더 설정 그대로 두고, 이번 푸시로 자동 재배포되는지 확인.
- 아직 Pages가 꺼져 있다면 → 켜기 전에 나한테 먼저 물어봐줘(어느 브랜치를 소스로 쓸지, `/`(루트)로 할지 `/docs`로 할지).
- 배포가 끝나면(GitHub Actions가 있다면 워크플로 상태까지) 최종 접속 URL을 알려줘.

## 5. 마지막 확인

- 배포된 사이트 URL을 알려주고, 로컬에서 `npx serve .`로 띄운 버전과 실제 배포 버전이 같은 커밋인지(즉 최신 변경사항이 실제로 반영됐는지) 확인해줘.
- 배포 후 브라우저에서 열었을 때 콘솔 에러가 없는지, 특히 `js/data.js`/`js/game.js`가 정상 로드되는지 확인해줘.

## 하지 말 것

- 원격 저장소가 연결 안 된 상태에서 임의로 새 저장소를 만들거나 강제 초기화하지 마.
- `--force` 푸시는 절대 하지 마(내가 명시적으로 요청하지 않는 한).
- Chapter2~5 관련 미완성 코드나 스크래치 파일을 실수로 커밋하지 마 — `git status`로 반드시 먼저 확인.
